# Killer-демо: design-артефакт, заземлённый орг-памятью (с/без RAG)

Кульминация лекции. Показывает, что RAG меняет **выход разработки** — и в итоге **код, который поедет в прод**, — а не просто отвечает на вопросы. Выход здесь — **design-документ** (проектное решение), из которого пишется реализация. Без орг-памяти агент проектирует фичу на **ошибочном допущении** о поведении платформы; с `search_org_memory` он достаёт задокументированное поведение и разворачивает решение в корректное.

## Связь с лекцией про SDD

Артефакт `design` и скилл `design` — те же, что в SDD-лекции (проектное решение → план → спецификация). SDD даёт **дисциплину** проектного решения; RAG делает его **умным** — заземляет шаг «2-3 подхода с трейд-оффами» и секцию «Рассмотренные альтернативы» в реальном поведении платформы и прошлых решениях, которые агент иначе бы не знал. Корпус RAG (kube-scheduler) целиком состоит из **design-артефактов** — KEP и design-proposal. Демо элегантно: RAG **над** design-доками → чтобы написать новый design-док, который станет кодом.

## Тема

Мы — **прикладная команда, которая пишет свой контроллер (оператор) поверх Kubernetes**. Планировщик мы не трогаем — мы **потребитель** его API. У важного пода **P** есть помощник **H** (локальный кеш), который должен работать **на той же ноде**, что и P. Наш контроллер следит за P и размещает H — это **наш код**, он пойдёт в реализацию.

Когда планировщик вытесняет поды ради P, он метит P на освобождённую ноду N (`status.nominatedNodeName`). Задача — спроектировать, **как и когда контроллер сажает H на ту же ноду**. Всё решает одно допущение: **гарантирует ли `nominatedNodeName` ноду.**

**Задокументированный факт (то, что должен достать RAG):**
- `nominatedNodeName` — лишь **подсказка**, а не гарантия: под идёт на обычное планирование заново; освободившееся место может занять другой высокоприоритетный под; жертвы умирают не мгновенно; P может оказаться на другой ноде.
- Значит, привязывать H к `nominatedNodeName` — гонка: H сядет на N один, а P уедет.

**Источник ретривала:** `kubernetes/design-proposals-archive/scheduling/pod-preemption.md` (семантика `nominatedNodeName`, разделы «Notes», «Preemption») + `keps/sig-scheduling/1923-prefer-nominated-node` (нода — предпочтение, а не бронь).

## Подготовка

```bash
# корпус должен включать design-proposals (пятый источник) — там семантика nominatedNodeName
uv run python scripts/download_designs.py

# скилл design уже лежит в .claude/skills/ этого репозитория
# зарегистрировать MCP-сервер в Claude Code
claude mcp add --transport stdio scheduler-memory -- uv run python mcp_server.py
```

## Сценарий (два прогона)

Запускай Claude Code **в этой папке** (тут и скилл, и MCP). Клонировать kubernetes не нужно — пишем design-документ, а не Go-код.

Задача агенту:

> «В нашем сервисе на Kubernetes важный под P и его помощник H (локальный кеш) должны работать на одной ноде. Мы пишем свой контроллер (оператор), который следит за P и размещает H, — это наш код, он пойдёт в реализацию. Планировщик Kubernetes при вытеснении метит P на освобождённую ноду через `status.nominatedNodeName`. Спроектируй, как и когда контроллер должен размещать H на той же ноде, что и P. Используй скилл `design`: разбери 2-3 подхода с трейд-оффами и секцией «Рассмотренные альтернативы» (Alternatives Considered), результат запиши в `docs/designs/`.»

- **Без `search_org_memory`** (сервер не зарегистрирован): агент считает `nominatedNodeName` гарантией и проектирует «прочитали `nominatedNodeName` P → сразу сажаем H на эту ноду». Код по такому design **ломается на гонке**: P не успел занять N, H сидит там один, P уехал на другую ноду.
- **С `search_org_memory`**: на шаге разбора подходов агент зовёт инструмент, достаёт из `pod-preemption.md`, что `nominatedNodeName` — **подсказка, а не гарантия** (со ссылкой), помечает наивный вариант в «Рассмотренных альтернативах» как отвергнутый и выбирает корректный: **ждать фактической привязки P к ноде (`spec.nodeName`) → тогда сажать H рядом; P переехал — H следом (reconcile).**

Порядок для сцены: сначала «без» (наивный design → баг в проде), потом «с» (контраст). Разные чистые сессии. Соль: **RAG изменил не только текст документа, но и код, который поедет в прод** — из багованного в корректный. И честно (врезка «худший для RAG случай»): Kubernetes публичен, на разовый вопрос это нашёл бы и агентский поиск по GitHub; настоящая ценность RAG — на разрозненной приватной памяти компании.

## Проверка ретривала

Что нужный фрагмент достаётся (иначе демо не сработает):

```bash
# семантика nominatedNodeName всплывает первой
uv run python retrieval.py "does nominatedNodeName guarantee the pod lands on that node" --mode hybrid --rerank --pile kep
uv run python evaluate.py --mode hybrid --rerank      # сравнить с --mode dense
```

Проверено: `search_org_memory` на этот вопрос возвращает `pod-preemption.md` с описанием `nominatedNodeName` как подсказки первым.

## Мост в SDD-лекцию

Здесь мы **сеем** артефакт `design` и скилл `design` (лекция 10, RAG). В лекции 12 (SDD) это разворачивается в дисциплину «проектное решение → план → спецификация». Связка: *орг-память заземляет проектное решение — RAG меняет design, а design меняет то, что построят.*
